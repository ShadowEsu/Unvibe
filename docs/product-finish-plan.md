# Uncode private-beta finishing plan

Last updated: 2026-07-15

Working branch: `codex/beta-core`
Primary surface: macOS Electron app in `app/`; `web/` is the shared backend and account dashboard.

This is a finishing plan, not a feature wish list. Status values are `Not started`, `In progress`, `Implemented — verification pending`, `Blocked`, and `Complete and tested`. A row is complete only when its stated completion criteria have been verified.

## P0 — Beta blockers

| ID | Problem | Affected files | User impact | Proposed fix | Dependencies | Tests required | Completion criteria | Status |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| P0-01 | Signing out only clears the desktop copy of the token; the server token remains usable. | `app/src/main/main.ts`, `app/src/main/backend.ts`, `web/src/data/*`, `web/app/api/v1/auth/signout/route.ts` | A token copied before logout can still access the account. | Add current-token revocation to the store contract, revoke it server-side, and make desktop logout report remote-revocation failures honestly while still clearing local state. | Existing opaque-token store. | Unit tests for MemoryStore revocation; route/auth integration test; old-token rejection check. | After logout, the old bearer token receives 401 and local identity is cleared while the identity-bound outbox remains recoverable. | Complete and tested (development store) |
| P0-02 | Desktop token storage falls back to reversible plaintext/base64 when macOS secure storage is unavailable. | `app/src/main/store.ts`, account UI | Device access can expose the session token while UI still permits account mode. | Fail closed for account sign-in when secure storage is unavailable; never persist a plaintext token; preserve local-only mode. | Electron `safeStorage`. | Token-vault tests; manual unavailable-keychain state. | No session token is written unless OS-backed encryption is available, and the user gets a recoverable message. | Implemented — manual keychain verification pending |
| P0-03 | Real AI review and comprehension endpoints accept anonymous requests. | `web/app/api/v1/reviews/route.ts`, `web/app/api/v1/comprehension/route.ts`, `app/src/main/review.ts`, `app/src/main/backend.ts` | Anyone can spend the configured provider credential and bypass account controls. | Require a valid bearer session whenever a real provider is active; allow anonymous mock responses only in development. Forward the desktop bearer token and surface 401/rate-limit errors. | Current auth store; AI provider selection. | Authorization-policy tests; local mock route check; real-provider authenticated check. | Real-provider calls without valid auth fail closed; authenticated streaming and quiz calls still work. | Implemented — real-provider verification blocked |
| P0-04 | Supabase operations ignore database errors and production can silently select the ephemeral MemoryStore. | `web/src/data/supabaseStore.ts`, `web/src/data/store.ts` | Writes can be reported as successful when lost; a production restart can erase all data. | Check and propagate every Supabase result; make MemoryStore an explicit development/test choice and expose a clear unavailable state when persistent configuration is absent. | Supabase environment configuration. | Store unit tests; missing-config production test; staging smoke test when credentials are available. | No failed database operation returns success; production cannot silently use volatile storage. | Implemented — Supabase verification blocked |
| P0-05 | Offline sync has a durable outbox but no scheduled exponential retry, visible status, manual retry, expired-auth recovery, or remote pull/reconciliation. | `app/src/main/sync.ts`, `app/src/main/store.ts`, `app/src/preload/preload.ts`, companion UI, `web/app/api/v1/history/route.ts` | Work may remain queued indefinitely and a second device or fresh login cannot restore remote history. | Add bounded exponential backoff with jitter, persisted/recoverable outbox semantics, sync status IPC, manual retry, auth-expired state, and idempotent remote merge by stable event ID. | P0-01 and reliable backend errors. | Online/offline/restart/duplicate/auth-expiry/multi-device tests. | Offline events survive restart, upload once after reconnect, UI reports state, and login reloads remote events without duplicates. | Implemented — full restart/offline staging verification pending |

| P0-07 | The required cloud-backed journey cannot be verified without a configured staging Supabase project and AI provider credential. | Runtime environment and staging deployment | A passing mock path does not prove real streaming, RLS, account persistence, or cross-device restore. | Complete local deterministic coverage, provide a staging checklist, then run the real journey only after the founder supplies/authorizes non-production credentials. | Founder-provided staging credentials; no production data. | Full 27-step journey; two-user RLS suite; restart/offline/package smoke test. | Evidence records every step against staging; otherwise the release remains not beta-ready. | Blocked |

## P1 — Required before inviting users

| ID | Problem | Affected files | User impact | Proposed fix | Dependencies | Tests required | Completion criteria | Status |

| P1-02 | Email verification, password reset, and refresh-token semantics are not implemented for the custom development email login; production uses Supabase device approval instead. | Auth routes and account UI | The UI can imply account capabilities that do not exist. | Keep development email auth disabled in production, present device activation as the supported beta method, remove unsupported recovery claims, and document session expiry behavior. | Founder decision if password login is desired later. | Production-config auth tests; copy/UI states. | Only supported auth paths are exposed and every path has loading/error/expiry handling. | Not started |
| P1-03 | Several companion navigation destinations are static feature descriptions or dead controls rather than connected beta features. | `app/src/renderer/companion/*` | Users see implied capabilities, fake progress surfaces, and buttons that do nothing useful. | Keep the beta navigation to connected surfaces; remove dead beta controls; make Home/Progress use only persisted evidence. | P0-05/P0-06. | Empty/offline/error/populated UI states; keyboard smoke test. | No dead button, fake metric, random percentage, or unlabelled mock remains in the visible beta navigation. | Implemented — manual UI verification pending |
| P1-04 | Account deletion is designed but has not been proven end to end against Supabase, including old-token rejection and cascade coverage. | account UI, `/api/v1/account`, stores, migrations | A user cannot trust that deletion removed all account data and sessions. | Keep typed confirmation, check every deletion result, rely on verified cascades/explicit deletes, revoke all tokens, wipe local state, and close widgets. | Staging Supabase; destructive test users only. | Dedicated disposable-user deletion integration test. | Remote rows are gone, local data/outbox is gone, widgets close, and all old tokens fail. | Blocked |
| P1-05 | Test-mode subscription infrastructure is absent on the latest core branch and pricing is not approved. | New additive billing migration/routes/UI when authorized | Pro cannot be tested safely and no entitlement can be enforced. | Implement provider-agnostic plan/customer/subscription/event records and server-side entitlements using Stripe test mode only; keep checkout hidden until test credentials and plan identifiers are approved. | Founder pricing decision; Stripe test credentials; webhook secret. | Idempotent webhook fixtures for all required transitions; entitlement tests. | Replayed test webhooks are idempotent, states are correct, and no live charge path exists. | Blocked |
| P1-06 | There is no verified, signed/notarized installable beta artifact or packaged-app auth test. | `app/package.json`, packaging docs/config, macOS permissions | External users may be unable to install or trust the binary. | Verify unsigned local arm64 packaging only as an engineering smoke test; document Intel support honestly; perform signing/notarization only after credentials are supplied. | Apple Developer signing/notarization credentials; approved backend URL. | Clean install, upgrade, offline startup, permission denial, auth callback, restart. | A signed/notarized artifact passes the checklist; until then external beta distribution remains blocked. | Blocked |
| P1-07 | RLS policies are plausible but have not been executed as a two-user isolation suite and omit explicit authenticated-role scoping/optimized ownership expressions. | `web/supabase/migrations/*`, new RLS test script | Backend security is asserted from SQL inspection, not proven. | Add an additive policy-hardening migration using `TO authenticated` and `(select auth.uid())`; run disposable User A/User B read/write tests in staging. | Staging Supabase credentials. | Cross-user projects/explanations/skills/streak/subscription/notes matrix. | Every forbidden operation fails in the actual database and permitted ownership operations succeed. | Blocked |

## Execution order

1. Repair authentication revocation and secure local token persistence.
2. Fail closed on backend persistence errors and protect paid AI endpoints.
3. Add durable sync retry/status/remote reconciliation.
4. Make streak and learning evidence portable and cautious.
5. Remove or park unconnected dashboard claims.
6. Run all local tests/builds, write `docs/beta-readiness-report.md`, checkpoint commits, and push only `codex/beta-core`.
7. Run credential-dependent staging/package checks only after the corresponding blocked inputs are provided.
